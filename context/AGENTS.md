# AGENTS.md (для разработки самого пакета)

Навигация по этому репозиторию для кодинг-агента, который РАЗРАБАТЫВАЕТ `@artefact/design-system` (добавляет компоненты/токены, правит сборку). Не про визуальный язык (это `DESIGN.md`) и не про перечень компонентов (это `COMPONENTS.md`) — про то, **в какой файл смотреть в какой момент** и по каким правилам устроен сам репозиторий.

Если задача — не разработка пакета, а его ИСПОЛЬЗОВАНИЕ в другом проекте (`npm install @artefact/design-system`) — этот файл не тот: смотри корневой `/AGENTS.md`, он публикуется вместе с пакетом и написан для потребителя, не для мейнтейнера.

## Это репозиторий ПАКЕТА, не приложения

`@artefact/design-system` — публикуемый npm-пакет (`npm run build` → `dist/index.js` + `dist/index.d.ts` + `dist/style.css`). Сборка целой страницы продукта («Pages») — не задача этого репозитория: здесь описываются и собираются только сами Foundation/Primitives/Components/Sections по отдельности, страницы продукта собирает потребляющее приложение, которое устанавливает этот пакет как зависимость.

## Три файла контекста — что читать когда

| Ситуация | Файл |
|---|---|
| Нужно значение токена (цвет/отступ/радиус/шрифт) или правило вида «сколько CTA можно на экране» | `context/DESIGN.md` |
| Нужно узнать, какой компонент уже есть, что он принимает, брать его или писать новый | `context/COMPONENTS.md` |
| Нужно понять, как всё это устроено как репозиторий (эта страница) | `context/AGENTS.md` |

Порядок при сборке чего-то нового: **`COMPONENTS.md`** (есть ли уже готовое) → **`DESIGN.md`** (какими токенами это красить) → код. Не наоборот — не изобретай значение, потом ищи, есть ли для него токен.

## Правило: код меняется → документы меняются в том же изменении

Это не «не забудь потом обновить документацию» — за один сеанс это забывается (см. историю с `scrim`/`accentGradient`, которые попали в код, но не в реестр вариантов, пока не хватились — тот же класс проблемы, только для документов, а не для реестра). Правило:

- **Новый/изменённый токен** (цвет, шаг типографики/отступа/радиуса) → добавь его в `context/DESIGN.md`'s frontmatter в том же изменении.
- **Новый/изменённый компонент** (Foundation/Primitives-свои/Components/Sections) → добавь строку в `context/COMPONENTS.md` в том же изменении.
- **Перед тем как считать задачу завершённой — прогони обе проверки:**
  ```
  npm run lint:design     # DESIGN.md внутренне непротиворечив
  npm run check:context   # DESIGN.md/COMPONENTS.md совпадают с кодом
  ```
  Обе должны завершиться с exit 0. Если `check:context` нашёл расхождение — это конкретный список того, что дописать, не общее «наверное что-то забыли».

### `npm run lint:design` — что проверяет и что в ответе — шум, а что — реальная проблема

Реальный CLI-линтер формата (`@google/design.md`, официальный пакет, не наша реализация) — детерминированно проверяет `context/DESIGN.md` сам по себе: битые `{colors.x}`-ссылки в `components`, WCAG-контраст пар `backgroundColor`/`textColor`, отсутствие обязательных полей, порядок секций, опечатки в ключах. Это НЕ проверка кода — линтер ничего не знает про `src/`.

Известные, принятые предупреждения (не чинить, если не появилась причина пересмотреть сам токен):
- `components.button-accent` — контраст 3.36:1 ниже общего порога 4.5:1. Линтер не различает размер текста; 3.36:1 проходит WCAG AA для крупного/полужирного текста кнопок (порог 3:1) — задокументированная причина в `index.css` рядом со значением, не случайность.
- Десятки `'X' is defined but never referenced by any component` — `components` во frontmatter — намеренно небольшая репрезентативная выборка (не вся система процедурно повторена туда), полный учёт использования цвета — это `context/COMPONENTS.md` + сам код, не этот блок. Не пытайся заткнуть все эти warnings искусственными `components`-записями без реального компонента за ними.

Единственное, что реально нужно чинить при появлении — **errors** (не warnings): например, `colors.*` обязаны быть настоящими CSS-цветами — градиенты туда не проходят, для них есть `components.<name>.backgroundColor` (см. `assistant-dock-btn` в файле).

### `npm run check:context` — свой скрипт, проверяет то, что линтер не видит

`scripts/check-context-sync.mjs` — написан для этого репозитория (не сторонний пакет): сравнивает реальные ключи `src/foundation/tokens.ts` (`color`/`spacing`/`radius`/`type`) с ключами в `context/DESIGN.md`, и реально экспортированные компоненты (`export function Name` в `src/foundation`, `src/primitives`, `src/components/composed`, `src/sections`) — с упоминаниями в `context/COMPONENTS.md`. **Не проверяет** shadcn-вендорные `components/ui/*` — они меняются редко и осознанно (через `npx shadcn add`, руками, не органическим ростом), поэтому их учёт в каталоге проверяется глазами при добавлении, а не автоматически.

Известные, документированные исключения из проверки токенов — список `KNOWN_COLOR_EXCEPTIONS` в начале скрипта (сейчас там ровно один: `accent-gradient` — живёт в `components`, а не в `colors`, по причине выше). Если заводишь новое намеренное расхождение такого рода — добавь его туда с комментарием-причиной, а не игнорируй красный exit code.

## Иерархия тиров — строгая, однонаправленная

```
Foundation  →  Primitives  →  Components  →  Sections   |   Pages (в потребляющем репо)
(src/foundation)  (src/primitives)  (src/components)  (src/sections)
```

Каждый тир импортирует только из тиров НИЖЕ себя, никогда выше и никогда вбок минуя уровень. `Card` — не монолит, это `Box` + фон из Primitives-тира; `Toolbar` — Radix-поведение (Primitives) + `Flex` (Foundation). Если компонент из `components/composed/*` тянет что-то из `sections/*` — это инверсия иерархии, чинить сразу, не откладывать.

`Sections` — намеренно тонкий тир, а не «вторая копия Components»: bespoke-«органы», для которых есть повторяющийся прецедент на нескольких реальных экранах продукта, а не любой странично-специфичный блок. Сейчас в `Sections` ровно два таких компонента (`SidebarNav`, `AssistantDock`) — оба явно параметризованы (`items`, `clientLabel`, `intro`, `fallbackAnswer` — без дефолта на конкретного клиента/продукт), потому что этот пакет обслуживает несколько продуктов Артефакта, а не один. Добавляя следующий Section, сначала проверь, что для него реально есть прецедент повторения В ЭТОМ ПАКЕТЕ (не в одном потребляющем приложении), а не просто удобно вынести из одного проекта.

## Честность содержимого — не изобретай данные, не привязывайся к одному клиенту

Ни один компонент/дефолт в этом пакете не должен молча подразумевать какого-то одного конкретного клиента/продукт Артефакта — обязательные пропсы (`SidebarNav`'s `items`) или явно-без-дефолта placeholder-тексты (`AssistantDock`'s `intro`/`fallbackAnswer`) — вместо того, чтобы «просто сработало» с чьим-то конкретным брендингом по умолчанию. Если чего-то пока нет в системе — это находка для `COMPONENTS.md` (пометить как пробел), а не повод имитировать похожими на правду данными.

## Реестры вместо хардкода

Публичные реестры (`SURFACE_VARIANTS` в `foundation/surface.tsx`, `ICON_NAMES`/`REGISTRY` в `primitives/Icon.tsx`, `semanticRoles`/`chartKeys` в `foundation/tokens.ts`) существуют, чтобы любой потребитель мог перечислить «всё, что есть» программно, а не полагаться на кем-то вручную продублированный список, который тихо расходится с кодом (именно так `scrim`/`accentGradient` в своё время остались незамеченными, пока их не хватились). Правило: любой новый вариант/токен добавляется В РЕЕСТР, а не только «используется где-то в JSX» — иначе он невидим для любого потребителя, который правильно читает реестр, а не угадывает.

**Исключение, не баг**: shadcn-вендорные компоненты на `class-variance-authority` (`Button`/`Badge`/`Alert`) физически не могут отдать такой реестр — `cva()` не возвращает никаких метаданных в рантайме (проверено: `Object.keys()` на результате `cva()` — пусто). Их варианты документируются в `COMPONENTS.md` текстом, руками — форкать vendor-исходники, чтобы это исправить, здесь запрещено («stay synced, not fork»).

## Verify against a reference, не на глаз

Если сверяешь вёрстку с каким-либо референсом (макет, чужая реализация, скриншот) — сверяй через `getBoundingClientRect()`/`getComputedStyle()` в реальном браузере (Playwright/Chrome), а не визуальным сравнением скриншотов. Обе реальные ошибки, из-за которых появилось это правило (сайдбар на `left:32` вместо `left:8`; хайлайт тоггла в Toolbar не подсвечивался), нашлись только измерением, не глазами — на глаз обе выглядели «почти похоже».

## CSS-специфичность — источник повторяющихся багов

**Инлайн `style` на React-элементе побеждает внешний CSS-класс всегда**, независимо от специфичности CSS-селектора. Это уже дважды ловилось в этом коде:
1. `ToolbarToggleItem` — инлайновый `background:"transparent"` навсегда убивал `[data-state="on"]`-правило из `Toolbar.css`.
2. Почти повторилось при сборке кнопки `AssistantDock` — чуть не взяли `<Surface variant="plain">` (инлайновый прозрачный фон) там, где нужен был градиент, управляемый снаружи через CSS-класс; поймали до того, как это стало багом, переключившись на сырой `<button className="...">`.

Правило: если компонент должен управляться СНАРУЖИ через CSS-класс (не через пропы), не давай ему одновременно инлайновый `style` на то же CSS-свойство.

## shadcn — синхронизация, не форк

Правило проекта: визуальный вид достигается ТОЛЬКО темой (`index.css`) + одним override-слоем (`src/brand-overrides.css`, атрибутные селекторы `[data-slot="..."]`) — исходники `components/ui/*` не трогаются никогда. Это позволяет обновлять shadcn через `npx shadcn add` без потери правок. Если задача требует поведения, которого нет у стандартного shadcn-компонента — это повод написать новый Components-тир композит (см. `COMPONENTS.md`), не повод редактировать вендорный файл.

## Технические заметки

- Алиас `@/` → `src/` (см. `tsconfig`/`vite.config.ts`) — импорты вида `@/foundation`, `@/primitives/Icon`, `@/components/ui/*`.
- **Репозиторий без git.** Откат — только через явный бэкап (`tar`), если делался; по умолчанию его нет. Действуй аккуратнее, чем в репозитории с git — нет «просто откатить commit».
- Журнал решений между сессиями (не про дизайн-систему — про сам процесс работы) лежит ВЫШЕ этого репозитория: `design-pipeline/.claude/HANDOFF.md` (родительская папка, не `shadcn-experiment/.claude/`). Если ищешь, почему что-то устроено именно так, а объяснения нет ни здесь, ни в коде — смотри туда.
