# context/

Файлы в этой папке — не документация в обычном смысле, а **контекст, рассчитанный на то, что его будет читать кодинг-агент** (в том числе внутри одной сессии, между сессиями, или другой агент/человек в новом репозитории). Это отдельный слой от кода: код (`src/`) — рабочая реализация, файлы здесь — описание того, что в ней есть и почему.

Три файла, три разных вопроса:

| Файл | Вопрос, на который отвечает | Статус |
|---|---|---|
| `DESIGN.md` | какие значения (токены) и правила визуального языка? | есть |
| `COMPONENTS.md` (манифест) | какие компоненты уже есть, что принимают, когда брать/не брать? | есть |
| `AGENTS.md` | в какой файл смотреть в какой ситуации при сборке новой страницы? | есть |

## Как устроен `DESIGN.md`

Формат — открытая спецификация [`design.md`](https://github.com/google-labs-code/design.md) от Google Labs («A format specification for describing a visual identity to coding agents»). У него ровно два слоя в одном файле:

1. **YAML frontmatter** (между двумя строками `---`) — машиночитаемые токены с точными значениями: `colors`, `typography`, `rounded`, `spacing`, `components`. Это то, что агент может распарсить и использовать буквально — без интерпретации.
2. **Markdown-тело** ниже — обычная проза, объясняющая *почему* значения такие и *как* их применять (например: «`accent` — редкость, одна на экран» — это правило, не токен, YAML такое не выразит).

Официальный порядок секций в теле (используются по необходимости, не все обязательны): `Overview → Colors → Typography → Layout & Spacing → Elevation & Depth → Shapes → Components → Do's and Don'ts`. Мы следуем этому порядку — если добавляешь новую секцию, вставляй её на своё место в этой последовательности, а не в конец файла.

### Где хранится правда, а где — только отражение

`DESIGN.md` **не является источником значений** — он их описывает. Реальный источник — код:

```
src/index.css              — сырые CSS-переменные (сами значения)
src/foundation/tokens.ts   — типизированные ссылки на них
src/brand-overrides.css    — как эти значения применяются К shadcn-компонентам
        ↓
context/DESIGN.md          — то же самое, но в формате для агента + правила
```

Если значение в коде меняется, `DESIGN.md` нужно поправить руками — правка не автоматическая (в отличие от Storybook-историй, которые читают `tokens.ts` напрямую в рантайме — см. `AGENTS.md` про self-generating stories). Это осознанный трейд-офф: `DESIGN.md` должен быть читаемым автономно, без подключения самого проекта, поэтому он не может быть просто рантайм-экспортом. Но то, что правка не автоматическая, не значит, что расхождение остаётся незамеченным — см. ниже.

### Две проверки — линтер формата + свой чек на дрифт с кодом

```
npm run lint:design     # DESIGN.md сам по себе корректен (официальный CLI-линтер формата)
npm run check:context   # DESIGN.md/COMPONENTS.md совпадают с реальным кодом (наш скрипт)
```

`lint:design` — официальный пакет `@google/design.md`, 9 правил формата: битые `{colors.x}`-ссылки, WCAG-контраст, порядок секций и т.п. Он ничего не знает о `src/` — только о внутренней непротиворечивости самого файла.

`check:context` (`scripts/check-context-sync.mjs`) — написан для этого репозитория: сверяет реальные токены (`tokens.ts`) и реально экспортированные компоненты (Foundation/Primitives-свои/Components/Sections) с тем, что описано в `DESIGN.md`/`COMPONENTS.md`. Это и есть автоматическое обнаружение дрифта — полное описание правила и известных исключений в `AGENTS.md`.

### Что здесь **сознательно не описано**

- Полный список компонентов и их API — это `COMPONENTS.md`, не `DESIGN.md`. `DESIGN.md` — про токены и правила уровня Foundation, не про конкретные React-компоненты.
- Как агенту навигировать по репозиторию и в каком порядке читать эти три файла — это `AGENTS.md`.
- Иерархия Foundation→Primitives→Components→Sections и правило «Storybook — витрина, не источник правды» — тоже в `AGENTS.md`, это про устройство репозитория, а не про визуальный язык.
