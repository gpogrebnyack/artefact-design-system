# COMPONENTS.md — каталог

Что здесь, а что в `DESIGN.md`: `DESIGN.md` — токены и правила визуального языка (Foundation-уровень). Этот файл — конкретные React-компоненты: что это, какой API, когда брать, когда — нет. Значения (цвета/отступы/радиусы) не повторяются здесь — они указаны по имени токена, расшифровка — в `DESIGN.md`.

Источник правды — код (`src/foundation`, `src/primitives`, `src/components`, `src/sections`). Этот файл описывает его состояние на момент написания; если он расходится с кодом — прав код, этот файл нужно поправить.

**Обозначения тира:**
- 🟢 **своё** — написано в этом проекте, полный контроль над API.
- 🔵 **shadcn/vendored** — стандартный shadcn-компонент (`components/ui/*`),
  используется как есть + иногда токен-оверрайд. Полный список пропсов — в
  документации shadcn (не переписываем её здесь, она устареет быстрее кода).

**Канон сторей:** стори различаются СОСТОЯНИЕМ/вариантом, не датасетом (два списка с разными данными — одна стори); имя стори — по функции/состоянию, не по контенту демо (`WithValue`/`Disabled`/`TitleOnly`, а не `Roster`/`Periods`/`TeamAverageMetricCard` — вычищено разом по всем блокам); проп-передатчик Foundation-шкалы (`radius`, `p`, `gap`) не получает собственной стори — шкала документируется один раз, в своей Foundation-галерее.

**Канон имён:** имя в Storybook = имя экспорта = имя в этом файле. Никаких «дружелюбных» псевдонимов (`Modal`, `Menu`, `Divider`, `Notification`, `EmptyState` — так эти компоненты назывались в Storybook раньше; теперь они `Dialog`, `DropdownMenu`, `Separator`, `Toaster`, `Empty`). Ищешь то, что импортируешь — находишь под тем же именем.

**Два жанра таблиц** (и только два): таблица-ВЫБОР — вход по задаче, колонки `Задача | Бери | Почему` (семейство Card, «Пилюли и теги»); таблица-КАТАЛОГ — вход по компоненту, колонки `Компонент | API | Когда брать / не брать` (все остальные). Новая таблица обязана взять одну из этих двух схем.

**Как организован Storybook:** семейства с внутренним выбором лежат стопочками (`Components/Card/*`, `Components/Chart/*`, `Components/Chat/*`, `Foundation/Layout/*`) — если тебе нужна «карточка чего-то», «график чего-то» или «как сверстать», открой папку семейства и выбери внутри. Всё, у чего семейства нет, лежит плоско. `Foundation/Layout/*` следует индустриальному стандарту категории «Layout» (Polaris «Layout and structure», MUI/Chakra «Layout»): страница на компонент — `AppShell` (каркас страницы; его стори `PageAnatomy` — «рецепт», все три высоты лейаута вложены как на реальной странице), `TitledRow` (строка-секция), `Box`/`Stack`/`Flex`/`Grid`/`Container` (раскладка внутри блока).

## Foundation

Чистое поведение, без цвета/идентичности. Всё выше строится на этом.

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Box` 🟢 | Базовый контейнер: `p/px/py`, `radius`, `width/maxWidth/minWidth`, `grow`. Всё остальное (включая `onClick`, `aria-*`) прокидывается на элемент напрямую. | Всегда, когда нужен `<div>` с паддингом/радиусом из шкалы — никогда не пиши `style={{padding: 16}}` вручную. |
| `Flex` 🟢 | `Box` + `display:flex`, `direction`, `gap`, `align`, `justify`, `wrap` (по умолчанию `true` — оборачивается сам). | Ряд элементов, которым можно перенестись на новую строку. |
| `Stack` 🟢 | `Flex` с `direction="column"` по умолчанию. | Вертикальная колонка — самый частый кейс, есть отдельным именем, чтобы не писать `direction="column"` каждый раз. |
| `Grid` 🟢 | `Box` + `display:grid`, `auto-fit`/`minmax` реflow по `minColWidth` (default 280) ИЛИ фиксированное число `columns`. `width:100%` и `align-content:"start"` пришиты намеренно. | Сетка карточек, которая сама решает, сколько колонок влезает — не считай breakpoints руками. **Разгаданный фантомный баг (важно для потребителей):** блочный элемент с `width:auto` внутри колоночного `Stack`/`Flex`, содержащий auto-fit грид, получает в Chromium высоту, измеренную ДО растяжения по ширине — грид в этот момент однокол­оночный и высоченный — и она не перемеряется: контейнер оказывается на ~750–1000px выше своего контента, пустой блок после сетки. `Grid` теперь несёт `width:100%` сам, но **если оборачиваешь `Grid` своим блочным div внутри `Stack` — дай обёртке `width:100%`** (готовый пример — `CollapsibleGroup`). Три cold-start сессии ловили этот баг и обходили самодельным Flex+ResizeObserver — больше не нужно. |
| `Container` 🟢 | `Box` + `max-width` (default 1280) + `margin-inline:auto`. | Центрированная колонка контента заданной ширины. |
| `Surface` 🟢 | `Box` + один из 5 background-вариантов (`glass/paper/muted/plain/scrim`, реестр `SURFACE_VARIANTS` в `surface.tsx`; `glass` — только 60%-заливка БЕЗ блюра, blur есть лишь у `scrim` — правило оверлеев, `DESIGN.md` → Elevation) + `interactive` — ховер кликабельной поверхности (подъём −2px + тихая заливка shadcn'овским hover-токеном `--accent`-слота, surface.css; accent-рамку из `.ecard:hover` источника попробовали и отклонили на ревью — оранжевая рамка на каждом ховере кричит). | Любая поверхность с фоном из системных токенов. Кликабельная карточка ОБЯЗАНА получить `interactive`: `onClick` без ховер-фидбэка читается как сломанный («выглядит кликабельным → обязано реагировать», DESIGN.md). Но это язык именно КАРТОЧЕК-СУЩНОСТЕЙ — широкий баннер/сводка ховер не получают даже с действием внутри (у источника `.vbanner` без hover): там действие несёт кнопка, а не вся поверхность. См. `DESIGN.md` → Elevation про разницу `glass` и `scrim`. |
| `TitledRow` 🟢 | Секция title-rail + main + опциональный side-rail (`rail`/`gap` в px). Настоящий CSS (`row.css`), не инлайн-стили — брейкпоинт ~1120px требует media query. **Правый рейл резервируется даже пустым** — main-колонка одной ширины на всех строках страницы; схлопнуть рейл — только явным `fullWidth` (это `.srow.no-side` из dashboard). Старое поведение «нет side → main шире» дало две разные ширины секций на одной странице — реальный баг, пойманный пересборкой источника. | **Правило из исходных страниц (35+12 реальных использований): левый рейл-заголовок — только когда секция группирует ≥2 отдельных карточек под одной шапкой** (например, «Операции» над сеткой из трёх карточек). Секция из ровно одной самодостаточной карточки рейл не получает — у карточки свой внутренний `<h3>`, и `TitledRow` вокруг неё не нужен вовсе. |
| `AppShell` 🟢 | `{sidebar, children}` → флекс сайдбар + колонка `max-width:1280, padding:26px 24px 90px`. Инвариант для всех страниц продукта, не одноразовое решение. | Корень любой реальной страницы. Не пересобирай эти значения руками на уровне страницы — это ровно тот баг, который уже был (сайдбар прилипал не туда). |

## Primitives

### Свои (полный контроль)

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Icon` | `<Icon name="..." size weight color label />`. Имя — ключ из governed-реестра `REGISTRY` (Phosphor-глифы; полный список в `ICON_NAMES`). `weight` типизирован как `"regular" \| "fill"` — остальные четыре стиля Phosphor (thin/light/bold/duotone) не пройдут компиляцию. | Всегда через `name`, никогда не импортируй Phosphor-глиф напрямую в компонент выше — тогда список иконок остаётся одним управляемым словарём. Если нужной иконки нет в `REGISTRY` — добавь её туда, не инлайни SVG на месте использования. Какой вес когда — правило в `DESIGN.md` → Shapes: `regular` — вся контурная иконография (дефолт), `fill` — только глифы-знаки (play/pause, маркер приоритета на чипе, ✦ у `SparkLink`). |
| `Image` | `<Image src alt width height fit radius loading />`. Настоящий пробел ни у Radix, ни у shadcn (Avatar покрывает только круглый fallback-кейс, не фото/иллюстрацию). | Любое `<img>` в продукте. `alt` обязателен — decorative-картинка получает `alt=""` явно, а не отсутствие атрибута. |
| `Text` | `<Text as size color weight align truncate />`. `size` — шаг шкалы типографики (`footnote…display`), резолвится в px+lineHeight из `tokens.ts`. | Любая строка текста в компоненте выше должна идти через `Text`, не через сырой `<span className="text-sm">` — единственное место, которое знает, что означает `caption` в px. `weight` не запечён в шаг шкалы — ставь явно (600 — единственное другое реально самохостящееся начертание, см. `DESIGN.md`). |

### shadcn/vendored (стандартный API — см. shadcn-документацию)

`Avatar`, `Badge`, `Button`, `ButtonGroup`/`ButtonGroupText`/`ButtonGroupSeparator`, `Checkbox`, `Collapsible`, `Input`, `Textarea`, `Label`, `Popover`, `RadioGroup`, `ScrollArea`, `Select`, `Skeleton`, `Switch`, `Toggle`/`ToggleGroup`, `Tooltip`, `Progress`, `Separator`, `Spinner`, `Dialog`.

Заметки по недавно довендоренным:
- `Skeleton` — загрузочный плейсхолдер. Повторяй ФОРМУ замещаемого контента (круг аватара + две строки), не одну серую простыню.
- `Collapsible` — база для сворачиваемых групп (поведение `.addr-group` из исходной страницы; две consumer-сессии успели написать этот стейт руками). Собранный паттерн — `CollapsibleGroup` ниже.
- `Popover` — кликом открываемый интерактивный флоат. Для hover-подсказки — `Tooltip`, не он.
- `ScrollArea` — управляемый скролл-контейнер вместо самодельного `maxHeight + overflow:auto` (реальный hand-roll из consumer-сессии).

**Осознанно НЕ вендорим** (проверяли — причина есть у каждого): `sidebar` (наш `SidebarNav` — bespoke под бренд, shadcn'овский — другая большая layout-система), `form` (есть `FormField`), `message`/`bubble` (есть свои Chat-компоненты по дизайну источника), `calendar`/`carousel`/`resizable`/`input-otp`/`slider` (ни одного реального кейса в продукте — «не строим наперёд»).

Используются как есть, без правки исходников. Наши оверрайды — на уровне темы/атрибутов, не пропсов:
- **Где проходит граница «сырых» tailwind-классов**: `className` на слоте vendored-компонента (`TableCell className="text-right"`, `TabsContent`, `CardContent`) — легитимный shadcn-идиом, это их контракт. А вот свой `<span>`/`<div>`/`<label>` с `text-*`/`flex`-утилитами — обход Foundation: текст идёт через `Text`, раскладка — через `Flex`/`Stack`/`Grid` (вычищено по всем сторям разом; не заноси обратно).
- `[data-slot="button"]`/`[data-slot="input"]`/`[data-slot="select-trigger"]` — высота/радиус пересчитаны в `brand-overrides.css` (см. `DESIGN.md` → Components).
- Если конкретному месту нужно другое поведение (например, `Select`, который умеет очищаться крестиком) — это уже не «просто shadcn Select», это новый Components-тир композит (см. `FilterSelect` ниже), а не проп на самом `Select`.
- **`ButtonGroup` — тут наш pill-оверрайд явно выключен.** `[data-slot="button"]` обычно всегда pill (`border-radius: 9999px`), но кнопка внутри `ButtonGroup` — исключение (`brand-overrides.css`, селектор с двумя `[data-slot]`): иначе pill-радиус на каждой кнопке перебивает `ButtonGroup`'ы собственные `rounded-l-none`/`rounded-r-none`, и сегментированный контрол просто не получается — так и было до этой правки (проверялось вживую при первой же попытке отрендерить компонент). Если понадобится третье радиус-исключение — паттерн такой же: два `[data-slot]` в селекторе побеждают одиночный независимо от порядка подключения файлов.
- **`Spinner`** — единственное намеренное исключение из правила «иконки только через `Icon`/`REGISTRY`»: это `Loader2Icon` из `lucide-react`, не Phosphor — фиксированный анимированный глиф, не элемент управляемого словаря.
- **`variant="outline"` у shadcn-компонентов по умолчанию рисует границу `border-border`, а `color.border` в этой системе прозрачен намеренно** (то же правило, что у осей чартов, см. `BarChart` ниже) — то есть outline без явного оверрайда невидим на любой поверхности темнее чистой страницы (карточка/`glass`/`paper`). **`Button`** — уже исправлено (`brand-overrides.css`: `[data-slot="button"][data-variant="outline"]` красится в `color.input`, ту же «видимую» линию, что у `Separator`) — outline-кнопка видна из коробки. **`Item`** — пока не исправлено на этом же уровне: нужна видимая обводка/подложка ряда — бери `variant="muted"`, не `outline`.

## Components

Primitives + Foundation, собранные в конкретную повторяемую сущность. Раздел организован по семействам — по вопросу «что мне нужно показать», а не по происхождению кода.

### Семейство Card — «карточка чего-то»

В Storybook — папка `Components/Card/*`. В коде это НЕ один компонент: `Card` — вендорная оболочка, остальные — независимые композиции на ней (склеивать их в один compound-API нельзя — `card.tsx` вендорен и не редактируется).

| Задача | Бери | Почему |
|---|---|---|
| Универсальная оболочка (Header/Content/Footer) | `Card` 🔵 | База всего семейства. На тёплом фоне страницы осознанно выбирай между ней и `Surface variant="glass"` (см. гайдлайн ниже). |
| Совет/инсайт в side-rail | `AdviceCard` 🟢 `icon, tone("ink"\|"accent"), value?, text, note?` | Для `TitledRow`'s `side`. `tone="accent"` — редкость (новый рекорд), не дефолт. Не путать с `Alert` (плоское уведомление, без слота под значение) и `Empty` (про отсутствие данных). |
| Карточка-метрика (число + тренд) | собери сам: `Card` + `Text` + `Icon` + `Sparkline` (строчный вариант без карточки — `MetricRow`) | Не формализована намеренно: один прецедент-источник (dashboard), правило «не строим наперёд». Держали демо-стори «скопируй как образец» — убрана на ревью (не-компонент в сайдбаре путал); повторится у второго-третьего реального потребителя — кандидат в экспорт. |
| Карточка с графиком (заголовок + легенда/стат + чарт) | `ChartCard` 🟢 `title, sub?, aside?, children` | Самая частая композиция dashboard-страницы (11 повторов в источнике). Внутрь — любой `Components/Chart/*` как children. Не собирай Card+заголовок+чарт руками. |
| Двухслойная «факт → совет» (наблюдение на бумаге + рекомендация на тинте) | `LayeredCard` 🟢 `children, footer?` | Анатомия `.rec`/`.up4-card` страницы разбора диалога: opaque-бумага с фактом внутри тонированной обёртки (rgba(0,0,0,.03), 8px рамка), совет/скрипт — в `footer` на тинте. Обе секции («Что улучшить» и «Допродажи») — это она; не собирай вложенные фоны руками. |
| Карточка сущности (аватар+имя+тренд+теги+метрики+CTA) | гайдлайн ниже | Не компонент — но правила сборки жёсткие, см. ниже. Ингредиенты теперь все готовые: `SemanticAvatarFallback` + `ScorePill` + `StatusBadge` + `MetricRow` + `Separator` + `Button`. |

**Гайдлайн: карточка сущности.** Паттерн собирался вручную уже в трёх независимых сессиях, каждый раз чуть по-разному. Правила:
- Разделитель — между КРУПНЫМИ смысловыми блоками, не перед каждым элементом. Три блока, не пять: (шапка + статус-тег вместе, это одно «сейчас с этим человеком») → (метрики) → (CTA). Тег после шапки — без своего отдельного `<Separator />`, это тот же блок, ему нужен только `gap`. Живая ошибка: агент поставил разделитель перед каждым элементом — 3 линии в одной небольшой карточке, шум вместо структуры. **И обратное правило: разделитель существует только МЕЖДУ двумя непустыми блоками** — если у карточки нет метрик (управляющий без цифр), их разделители уходят вместе с ними; две линии вокруг пустой зоны — не «сохранение структуры», а мусор (живая ошибка из пересборки источника). Равнять высоты карточек пустыми зонами тоже не нужно — `Grid` с `align:start` этим управляет сам.
- `Surface variant="glass"` даёт смешанный с фоном кремовый тон; `variant="paper"` более opaque/белый — осознанный выбор, не default по инерции.
- **Имя человека — всегда `body`/600, где бы он ни показывался** (карточка, компактная строка инсайтов, список): источник держит один размер (`--text-body`) и в `.ename`, и в `.iline .nm`. Не даунскейлить имя до `caption` в «маленьких» рядах — компактность даёт вторая строка (`footnote`), не усушка имени.
- Тон аватара и тон бейджа — из одной роли, и оба — **мягкая пара** (`color.<tone>Soft`/`SoftForeground`): так красит свои аватары сама исходная страница (`.eava`). *Исправление прежней версии этого правила* — «яркая пара на аватаре» была изобретением одной из consumer-сессий, а не правилом источника; пересборка komanda.html это вскрыла. Яркая пара (`color.<tone>`) — язык score-гейджа (`RadialChart`), не аватара.

### Семейство Chart — все графики

В Storybook — папка `Components/Chart/*`. Все — настоящий Recharts через shadcn's `ChartContainer`; grid/оси — `color.input`, никогда `color.border` (тот прозрачный по правилу).

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `BarChart` / `LineChart` / `AreaChart` | `data, xKey, series: {key,label,color}[], height?, showLegend?` (+`stacked?` у Area) | Базовые типы. `showLegend` по умолчанию `series.length > 1`. |
| `DonutChart` | `data: {key,label,value,color}[], height?, showLegend?` | Категориальная разбивка (Pie, `innerRadius:"60%"`). |
| `DistributionChart` | `groups: {name,total,segments:{label,count,widthPct}[]}[], rowHeight?` | Горизонтальный stacked-bar на группу; недостающие до 100% сегменты дорисовываются приглушённым `rest`. |
| `ScatterChart` | `series: {key,label,color,data:{x,y}[]}[], xLabel?, yLabel?, height?, showLegend?` | Корреляция двух числовых величин. |
| `RadialChart` | `value(0–100), color, trackColor?, size?, label?` | Круговой гейдж («ScoreRing» — пересобирался с нуля на каждом экране, пока не был формализован). |
| `ComposedChart` | `data, xKey, series: {key,label,color,type:"bar"\|"line"\|"area",axis?:"left"\|"right"}[], height?, showLegend?` | Смешанные типы серий. **`axis:"right"` не декоративен**: серии на сильно разных масштабах на общей оси сплющиваются в прямую линию. |

### Тулбары и сегмент-контролы — какой когда

Три компонента дают визуально похожий «сегментированный» ряд — роли разные:

- **Переключение состояния/периода** (Часы/Дни/Недели, вкладкоподобное) → `ToolbarToggleGroup` + `ToolbarToggleItem` (roving focus, aria-состояние «выбрано»); голый примитив под ним — `ToggleGroup`.
- **Ряд независимых действий-кнопок** (два спаренных icon-баттона, сортировка) → `ButtonGroup` (Primitives) — каждая кнопка делает действие, ничего не «выбрано».
- **Переход между разделами/страницами** → `NavigationMenu` (Components) — настоящие `<a href>` в `<nav>`-ландмарке: URL/history, открытие в новой вкладке, «navigation» для скринридера. Правило одной строкой: **меняешь данные на экране — ToggleGroup, меняешь экран — NavigationMenu**; «абстрактный» ToggleGroup вместо навигации теряет весь контракт ссылки.

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Toolbar` + `ToolbarGroup` + `ToolbarToggleGroup` + `ToolbarToggleItem` + `ToolbarSeparator` + `ToolbarButton` 🟢 | `Toolbar({orientation, bare})`; `ToolbarToggleGroup({value, onValueChange})` | На `radix-ui` Toolbar напрямую (у shadcn нет рецепта). **`bare=true` обязателен**, если внутри ровно одна самоокрашенная группа — иначе пилюля-в-пилюле (реальный баг, уже чинили). |

### Списки и композиционные оболочки (shadcn/vendored)

`Card` (см. семейство Card), `Empty`, `Tabs`, `Alert`, `AlertDialog`, `DropdownMenu`, `Pagination`, `Table`, `Toaster` (sonner), `Item`, `NavigationMenu`. Композиционные оболочки со слотами — поэтому Components, не Primitives. Граница с Primitives-vendored: `Dialog`/`Select`/`Popover`/`ScrollArea` остаются примитивами — многочастная разметка, но ОДНО взаимодействие (открыть модалку, выбрать одно, показать флоат, проскроллить); оболочки же структурируют контент СЛОТАМИ (Header/Content/Footer у Card, Media/Title/Actions у Item). Тест: если children кладутся внутрь как есть, без раскладки по слотам, — это примитив, какой бы многочастный DOM ни был у него внутри. `NavigationMenu` (8 слотов, выпадающие панели, свой viewport) переехал сюда именно по этому критерию — в Primitives он лежал по инерции «vendored = примитив». API — см. shadcn-документацию.

- **`Alert`** — статичная заметка (заголовок + текст), ось одна: `variant` (`default`/`destructive`). НЕ для продуктового баннера с кнопкой действия («Распознано N голосов» + «Разметить») — это рецепт `Surface variant="muted"` + `Button` (см. правило у `Surface`); и НЕ для «требуют внимания» по метрикам — это язык `accent`, а `destructive` только для жёстких ошибок (правило danger в `DESIGN.md`).
- **`AlertDialog`** — блокирующее подтверждение разрушительного действия. Ровно то место, куда указывает правило про `danger` в `DESIGN.md`: кнопка подтверждения — единственное легитимное место для danger-filled кнопки. Для нейтральных модалок — `Dialog`, не он.

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Item` + `ItemGroup`/`ItemSeparator`/`ItemMedia`/`ItemContent`/`ItemTitle`/`ItemDescription`/`ItemActions`/`ItemHeader`/`ItemFooter` | Governed строка списка: иконка/медиа + заголовок + описание + trailing-действия. `variant("default"\|"outline"\|"muted")`, `size("default"\|"sm"\|"xs")`. | Любой список строк одной формы — диалоги, уведомления, activity feed. Не собирай flex-строку + свой `<hr>` под этот же паттерн. `variant="outline"` — см. предупреждение про `color.border` выше. |
| `CollapsibleGroup` 🟢 `title, count?, extra?, defaultOpen?, open?, onOpenChange?, children` | Сворачиваемая группа с кликабельной шапкой: заголовок + счётчик + слот под бейдж + поворачивающийся шеврон (`.addr-group` из источника). | Группировка карточек по адресу/категории. Контролируемый режим (`open`/`onOpenChange`) — для «Свернуть всё». Две consumer-сессии писали этот стейт руками. Длинный список внутри — оберни в `ScrollArea`, не самодельный overflow. |

⚠️ **Известный пробел:** `Toaster` (sonner) использует `next-themes`, но `ThemeProvider` не смонтирован — тема тостов всегда резолвится в `"system"`. Не чинилось, низкий приоритет.

⚠️ **Известный пробел (баг апстрима shadcn, проверено в их registry):** `Progress` не передаёт `value` в Radix Root — визуально полоса правильная, но для скринридеров ЛЮБОЕ значение читается как indeterminate (нет `aria-valuenow`). Локально не патчим (vendored-исходники не редактируются); если критично на конкретном экране — ставь `aria-valuenow`/`aria-valuemax` пропами прямо на `<Progress>` (они долетят до Root через `...props`).
