# COMPONENTS.md — каталог

Что здесь, а что в `DESIGN.md`: `DESIGN.md` — токены и правила визуального языка (Foundation-уровень). Этот файл — конкретные React-компоненты: что это, какой API, когда брать, когда — нет. Значения (цвета/отступы/радиусы) не повторяются здесь — они указаны по имени токена, расшифровка — в `DESIGN.md`.

Источник правды — код (`src/foundation`, `src/primitives`, `src/components`, `src/sections`). Этот файл описывает его состояние на момент написания; если он расходится с кодом — прав код, этот файл нужно поправить.

**Обозначения тира:**
- 🟢 **своё** — написано в этом проекте, полный контроль над API.
- 🔵 **shadcn/vendored** — стандартный shadcn-компонент (`components/ui/*`),
  используется как есть + иногда токен-оверрайд. Полный список пропсов — в
  документации shadcn (не переписываем её здесь, она устареет быстрее кода).

**Канон сторей:** стори различаются СОСТОЯНИЕМ/вариантом, не датасетом (два списка с разными данными — одна стори); имя стори — по функции/состоянию, не по контенту демо (`WithValue`/`Disabled`/`TitleOnly`, а не `Roster`/`Periods`/`TeamAverageMetricCard` — вычищено разом по всем блокам); проп-передатчик Foundation-шкалы (`radius`, `p`, `gap`) не получает собственной стори — шкала документируется один раз, в своей Foundation-галерее.

**Канон имён:** имя в Storybook = имя экспорта = имя в этом файле. Никаких «дружелюбных» псевдонимов (`Modal`, `Menu`, `Divider`, `Notification`, `EmptyState` — так эти компоненты назывались в Storybook раньше; теперь они `Dialog`, `DropdownMenu`, `Separator`, `Toaster`, `Empty`). Ищешь то, что импортируешь — находишь под тем же именем.

**Два жанра таблиц** (и только два): таблица-ВЫБОР — вход по задаче, колонки `Задача | Бери | Почему` (семейство Card, «Пилюли и теги»); таблица-КАТАЛОГ — вход по компоненту, колонки `Компонент | API | Когда брать / не брать` (все остальные). Новая таблица обязана взять одну из этих двух схем.

**Как организован Storybook:** семейства с внутренним выбором лежат стопочками (`Components/Card/*`, `Components/Chart/*`, `Components/Chat/*`, `Foundation/Layout/*`) — если тебе нужна «карточка чего-то», «график чего-то» или «как сверстать», открой папку семейства и выбери внутри. Всё, у чего семейства нет, лежит плоско. `Foundation/Layout/*` следует индустриальному стандарту категории «Layout» (Polaris «Layout and structure», MUI/Chakra «Layout»): страница на компонент — `AppShell` (каркас страницы; его стори `PageAnatomy` — «рецепт», все три высоты лейаута вложены как на реальной странице), `TitledRow` (строка-секция), `Box`/`Stack`/`Flex`/`Grid`/`Container` (раскладка внутри блока).

## Foundation

Базовый слой, на котором строится всё выше: раскладка (`Box`/`Flex`/`Stack`/`Grid`/`Container`/`Surface`/`TitledRow`/`AppShell` — поведение без идентичности), типографика (`Text`/`Heading`) и токены (`DESIGN.md`).

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Box` 🟢 | Базовый контейнер: `p/px/py`, `radius`, `width/maxWidth/minWidth`, `grow`. Всё остальное (включая `onClick`, `aria-*`) прокидывается на элемент напрямую. | Всегда, когда нужен `<div>` с паддингом/радиусом из шкалы — никогда не пиши `style={{padding: 16}}` вручную. |
| `Flex` 🟢 | `Box` + `display:flex`, `direction`, `gap`, `align`, `justify`, `wrap` (по умолчанию `true` — оборачивается сам). | Ряд элементов, которым можно перенестись на новую строку. |
| `Stack` 🟢 | `Flex` с `direction="column"` и **`width:100%` по умолчанию**. | Вертикальная колонка — самый частый кейс. `width:100%` вшит: колонка почти всегда во всю ширину контейнера И это самозащита от Chromium phantom-height (обёртка `Stack` вокруг auto-fit `Grid` больше не даёт фантомный разрыв — см. `Grid`). Для редкого shrink-wrap-`Stack` в ряду задай `width` явно. |
| `Grid` 🟢 | `Box` + `display:grid`, `auto-fit`/`minmax` реflow по `minColWidth` (default 280) ИЛИ фиксированное число `columns`. `width:100%` и `align-content:"start"` пришиты намеренно. | Сетка карточек, которая сама решает, сколько колонок влезает — не считай breakpoints руками. **Гоча Chromium:** блочный элемент с `width:auto` внутри колоночного `Stack`/`Flex`, содержащий auto-fit грид, получает в Chromium высоту, измеренную ДО растяжения по ширине — грид в этот момент однокол­оночный и высоченный — и она не перемеряется: контейнер оказывается на ~750–1000px выше своего контента, пустой блок после сетки. `Grid` несёт `width:100%` сам, и `Stack` **теперь тоже** (дефолт), поэтому обычная обёртка «заголовок + `Grid`» через `<Stack>` безопасна из коробки. Гоча остаётся ТОЛЬКО если обернул `Grid` в СЫРОЙ `<div>` (не `Stack`/`Box`) внутри колоночного флекса — тогда дай тому div `width:100%`. |
| `Container` 🟢 | `Box` + `max-width` (default 1280) + `margin-inline:auto`. | Центрированная колонка контента заданной ширины. |
| `Surface` 🟢 | `Box` + один из 6 background-вариантов (`panel/paper/outline/muted/plain/overlay`, реестр `SURFACE_VARIANTS` в `surface.tsx`; `outline` — paper + hairline `--input`, ЕДИНСТВЕННАЯ легальная рамка-поверхность — для карточки на одноцветной подложке, где заливкам нечем разделять, см. DESIGN.md → «Разделение поверхностей»; `panel` — сплошной cream-1 (стекло упразднено), blur есть лишь у `overlay` — правило оверлеев, `DESIGN.md` → Elevation) + `interactive` — ховер кликабельной поверхности (подъём −2px + тихая заливка shadcn'овским hover-токеном `--accent`-слота, surface.css; НЕ accent-рамка — оранжевая рамка на каждом ховере кричит). | Любая поверхность с фоном из системных токенов. Кликабельная карточка ОБЯЗАНА получить `interactive`: `onClick` без ховер-фидбэка читается как сломанный («выглядит кликабельным → обязано реагировать», DESIGN.md). Но это язык именно КАРТОЧЕК-СУЩНОСТЕЙ — широкий баннер/сводка ховер не получают даже с действием внутри (у источника `.vbanner` без hover): там действие несёт кнопка, а не вся поверхность. См. `DESIGN.md` → Elevation про разницу `panel` и `overlay`. |
| `TitledRow` 🟢 | Секция title-rail + main + опциональный side-rail (`rail`/`gap` в px). Настоящий CSS (`row.css`), не инлайн-стили — брейкпоинт ~1120px требует media query. **Строковый `title` сам становится секционным `Heading`** (h2, `title`/`semibold` — на ступень ниже страничного h1 `headline`, см. DESIGN.md → Typography «ладдер заголовков»; иерархия размером, вес у всех semibold) — не передавай туда голый текст ожидая заголовок, но и не оборачивай сам; свой узел (`<Heading>`, стек с подзаголовком) переопределяет. **Правый рейл резервируется даже пустым** — main-колонка одной ширины на всех строках страницы; схлопнуть рейл — только явным `fullWidth` (это `.srow.no-side` из dashboard). | **Левый рейл-заголовок — только когда секция группирует ≥2 отдельных карточек под одной шапкой**. Секция из одной самодостаточной карточки рейл не получает. **Широкая «сцена» (плеер, `MomentStrip`, широкая таблица) — это `fullWidth`/`fullBleed`, НИКОГДА не строка с боковым рейлом:** рейл сжимает main, и широкий контент вылезает за колонку (проверено вживую — страница поехала на ~400px вбок). Инсайт к такой сцене клади отдельной строкой, а не в её `side`. |
| `AppShell` 🟢 | `{sidebar, children, gap?}` → флекс сайдбар + колонка `max-width:1280, padding:26px 24px 90px`, **которая сама раскладывает секции по вертикали** (`gap` по умолчанию `lg`). Инвариант для всех страниц продукта. | Корень любой реальной страницы. Клади секции (`PageHero`, `TitledRow`-ы) прямо в него — **НЕ оборачивай их в свой `Stack`** (это старый ад-хок; забыл обернуть — секции слипались). Другой ритм — через проп `gap`. |
| `Text` 🟢 | `<Text as size color weight align truncate />`. `size` — шаг шкалы (`footnote…display`) → px + lineHeight + letterSpacing из `tokens.ts`; `weight` — имя (`regular/medium/semibold`), не число. | Тело/проза/лейблы/**значения** (даже крупные жирные — счёт, цена: это данные, не заголовок). Не сырой `<span className="text-sm">`. Заголовок → **`Heading`** (правило двух корзин, `DESIGN.md` → Typography). |
| `Heading` 🟢 | `<Heading as size color weight align truncate />` — модель Radix Themes: тот же API и **та же полная шкала размеров**, что у `Text` (сетка НЕ урезана — размеры общие). Отличие от `Text` ровно в трёх вещах, как в Radix: элемент `h1–h6`, туже лидинг, тяжелее вес по умолчанию. Дефолты `as="h1"`, `size="headline"`, `weight="semibold"`; tight-лидинг (кэп 1.25) зашит. H1–H6 — через `as`, размер — через `size` (разведены). | Заголовок с ПРОИЗВОЛЬНЫМ размером (override). Для типовых уровней бери пресеты ниже — они зашивают ладдер, чтобы размер не разъезжался. Line-height руками не ставь. Выбор `Heading`/пресет vs `Text` — по РОЛИ (тайтл vs тело/значение), не по размеру: сетка одна. |
| `PageTitle` / `SectionTitle` 🟢 | Пресеты `Heading` с зашитым **ладдером уровней** (DESIGN.md → Typography): `PageTitle` — h1/`headline`(24)/semibold · `SectionTitle` — h2/`title`(20)/semibold. Все заголовки semibold — иерархия строится РАЗМЕРОМ, не весом. `size`/`as`/`weight` переопределяемы. Карточный заголовок (h3) — это `<Heading as="h3" size="subhead">` 18/semibold (имя `CardTitle` занято слотом shadcn `Card`). | **Заголовок уровня бери отсюда, а не `Heading` с ручным `size`** — тогда page(24)/section(20)/card(18) читаются как разные уровни. `TitledRow` строковый `title` → `SectionTitle` сам; `PageHeader` → `PageTitle`. Заголовок не делай легче родителя (подзаголовок жирнее секции = инверсия). |

### Сборка страницы — скелет

Реальная страница собирается всегда одинаково, снаружи внутрь:

1. **`AppShell`** — корень (сайдбар + контент-колонка). Секции клади прямо в него; он сам разводит их по вертикали. **Свой `Stack` вокруг секций не нужен.**
2. **Шапка страницы — готовая `PageHeader` ИЛИ `PageHero`, не собирай руками.** `PageHeader` — минимальная (title в рейле + мета-строка + один CTA), когда под шапкой сразу идут карточки. `PageHero` — богаче (крошки-назад над title, статус-пилюли под ним, `sub`-мета, `lead`-абзац), для списочных и детальных страниц. Правило одной строкой: **карточки впритык под шапкой → `PageHeader`; списочная/детальная страница → `PageHero`.**
3. **Лента `TitledRow`-секций.** Обычно у секции есть `title` (строка → авто-`h2`) в левом рейле + контент в main + **изредка** компактный инсайт в правом `side`-рейле (значение + `SparkLink`; не карточная сетка и не контролы — рейл ~180px, «заметка на полях»). **Врезок — 1–2 на страницу максимум, НЕ у каждой секции:** рейл это редкий акцент на главном инсайте, а не вторая колонка. Большинство секций рейл резервируют ПУСТЫМ — норма, не пропуск (см. DESIGN.md → Rail-конвенция). **Исключение (не противоречие с правилом ниже):** секция из ОДНОЙ самодостаточной карточки, у которой уже есть свой заголовок, в `TitledRow` вообще не заворачивается — рейл-тайтл нужен, когда шапка стоит НАД контентом секции (проза, набор строк, несколько карточек), а не дублирует заголовок единственной карточки.

Жёсткие правила, которые легко нарушить:
- **Контентная секция сидит на `Surface variant="panel"`** — остров на тёплом градиенте страницы. Это касается и аналитических блоков без собственной карточной обёртки (`ScoreHeatmap`, `WaveformPlayer`+`MomentStrip`, ряды метрик): их локальный тинт/бумага НЕ заменяют подложку секции — без неё блок висит голым на фоне и выпадает из ряда островов. Запись (плеер + лента моментов) — в ОДНОЙ панели на обоих, не по панели на компонент. Сами по себе такие блоки подложку не рисуют — это ответственность страницы.
- **Заголовки — только `Heading`/строковый `title`**, не голый `Text`/строка (иначе тайтл теряет и вид, и семантику h2). Значения (суммы, %) — `Text`, даже крупные (правило двух корзин, `DESIGN.md`).
- **Широкие «сцены»** (плеер, `MomentStrip`, широкая таблица, полноширинная лента) → `TitledRow fullWidth`/`fullBleed`, **без бокового рейла** — рейл сжимает колонку и контент вылезает за неё.
- **Обратное и важное: проза/сводка/вердикт — в main-колонке (зарейлена), НИКОГДА не `fullWidth`/`fullBleed`.** `fullWidth` — исключение только для широких ДАННЫХ/сцен, не для текстовой карточки. Сводка сверху страницы («пульс», «итог») — это обычный `TitledRow` с заголовком в рейле и карточкой в 832-main, как «Сводка команды» в `Pages/Komanda`, а НЕ full-width блок на всю ширину контента. Растянутая на всю ширину прозаическая карточка ломает «main одной ширины» и читается инородной простынёй.
- **Ширины/рейлы — из `TitledRow`/`AppShell`**, не хардкод-пиксели и не свои flex-basis. Main одной ширины на каждой строке — иначе страница расслаивается на два лейаута.

## Primitives

### Свои (полный контроль)

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Icon` | `<Icon name="..." size weight color label />`. Имя — ЛЮБОЙ глиф каталога Phosphor в kebab-case («calendar», «caret-left», …; весь каталог из 1512 глифов вкомпилен в пакет, полный словарь — тип `PhosphorIconName` / массив `PHOSPHOR_ICON_NAMES`, поиск — стори Primitives/Icon → CatalogSearch) ЛИБО семантический алиас системы (`dashboard`, `spark`, `voice`, … — карта `SEMANTIC_ALIASES` в `Icon.tsx`, галерея в `ICON_NAMES`). `weight` типизирован как `"regular" \| "fill"` (остальные четыре стиля Phosphor не вкомпилированы и не пройдут компиляцию), а его ДЕФОЛТ самоприменяющийся по размерному правилу DESIGN.md → Shapes: пиктограмма <16px → fill, служебные штрихи (caret/arrow/trend/check/x/точки) — regular всегда; явный `weight` побеждает. | Всегда через `name`, никогда не импортируй Phosphor-глиф напрямую в компонент выше. Для ПОВТОРЯЮЩЕЙСЯ смысловой роли добавь алиас в `SEMANTIC_ALIASES` (чтобы «какая иконка что значит» решалось в одном месте); разовый глиф — бери прямым именем каталога, алиас не заводи. Какой вес когда — правило в `DESIGN.md` → Shapes: `regular` — вся контурная иконография (дефолт), `fill` — только глифы-знаки (play/pause, маркер приоритета на чипе, ✦ у `SparkLink`). |
| `Image` | `<Image src alt width height fit radius loading />`. Настоящий пробел ни у Radix, ни у shadcn (Avatar покрывает только круглый fallback-кейс, не фото/иллюстрацию). | Любое `<img>` в продукте. `alt` обязателен — decorative-картинка получает `alt=""` явно, а не отсутствие атрибута. |

`Text`/`Heading`/`PageTitle`/`SectionTitle` — типографика, живёт в **Foundation** (таблица выше), не здесь.

### shadcn/vendored (стандартный API — см. shadcn-документацию)

`Avatar`, `Badge`, `Button`, `ButtonGroup`/`ButtonGroupText`/`ButtonGroupSeparator`, `Checkbox`, `Collapsible`, `Input`, `Textarea`, `Label`, `Popover`, `RadioGroup`, `ScrollArea`, `Select`, `Skeleton`, `Switch`, `Toggle`/`ToggleGroup`, `Tooltip`, `Progress`, `Separator`, `Spinner`, `Dialog`.

- **`Button`-варианты — `default` (тёмная ink-CTA), `secondary`, `outline`, `ghost`, `destructive`, `link`.** Дефолтная залитая кнопка у нас ТЁМНАЯ, не оранжевая — это осознанно.
- **Оранжевая акцентная CTA — это `AccentButton` 🟢** (Components), НЕ вариант `Button`: `<AccentButton>…</AccentButton>` с любыми пропсами `Button` (size/иконки/onClick). Одна на экран (DESIGN.md → «одна accent-кнопка»). Заливка `color.accent` + ховер `color.accentHover` зашиты — не собирай оранжевую кнопку инлайн-стилем сам.

Заметки по недавно довендоренным:
- `Skeleton` — загрузочный плейсхолдер. Повторяй ФОРМУ замещаемого контента (круг аватара + две строки), не одну серую простыню.
- `Collapsible` — база для сворачиваемых групп. Собранный паттерн — `CollapsibleGroup` ниже.
- `Popover` — кликом открываемый интерактивный флоат. Для hover-подсказки — `Tooltip`, не он.
- `ScrollArea` — управляемый скролл-контейнер вместо самодельного `maxHeight + overflow:auto`. Работает и с явным `height`, и с `maxHeight` (brand-overrides прокидывает ограничение во viewport).

**Осознанно НЕ вендорим** (проверяли — причина есть у каждого): `sidebar` (наш `SidebarNav` — bespoke под бренд, shadcn'овский — другая большая layout-система), `form` (есть `FormField`), `message`/`bubble` (есть свои Chat-компоненты по дизайну источника), `calendar`/`carousel`/`resizable`/`input-otp`/`slider` (ни одного реального кейса в продукте — «не строим наперёд»).

Используются как есть, без правки исходников. Наши оверрайды — на уровне темы/атрибутов, не пропсов:
- **Где проходит граница «сырых» tailwind-классов**: `className` на слоте vendored-компонента (`TableCell className="text-right"`, `TabsContent`, `CardContent`) — легитимный shadcn-идиом, это их контракт. А вот свой `<span>`/`<div>`/`<label>` с `text-*`/`flex`-утилитами — обход Foundation: текст идёт через `Text`, раскладка — через `Flex`/`Stack`/`Grid` (вычищено по всем сторям разом; не заноси обратно).
- `[data-slot="button"]`/`[data-slot="input"]`/`[data-slot="select-trigger"]` — высота/радиус пересчитаны в `brand-overrides.css` (см. `DESIGN.md` → Components).
- Если конкретному месту нужно другое поведение (например, `Select`, который умеет очищаться крестиком) — это уже не «просто shadcn Select», это новый Components-тир композит (см. `FilterSelect` ниже), а не проп на самом `Select`.
- **`ButtonGroup` — тут наш pill-оверрайд явно выключен.** `[data-slot="button"]` обычно всегда pill (`border-radius: 9999px`), но кнопка внутри `ButtonGroup` — исключение (`brand-overrides.css`, селектор с двумя `[data-slot]`): иначе pill-радиус на каждой кнопке перебивает `ButtonGroup`'ы собственные `rounded-l-none`/`rounded-r-none`, и сегментированный контрол просто не получается — так и было до этой правки (проверялось вживую при первой же попытке отрендерить компонент). Если понадобится третье радиус-исключение — паттерн такой же: два `[data-slot]` в селекторе побеждают одиночный независимо от порядка подключения файлов.
- **`Spinner`** — единственное намеренное исключение из правила «иконки только через `Icon`»: это `Loader2Icon` из `lucide-react`, не Phosphor — фиксированный анимированный глиф, не элемент управляемого словаря.
- **`variant="outline"` у shadcn-компонентов по умолчанию рисует границу `border-border`, а `color.border` в этой системе прозрачен намеренно** (то же правило, что у осей чартов, см. `BarChart` ниже) — то есть outline без явного оверрайда невидим на любой поверхности темнее чистой страницы (карточка/`panel`/`paper`). **`Button`** — уже исправлено (`brand-overrides.css`: `[data-slot="button"][data-variant="outline"]` красится в `color.input`, ту же «видимую» линию, что у `Separator`) — outline-кнопка видна из коробки. **`Item`** — пока не исправлено на этом же уровне: нужна видимая обводка/подложка ряда — бери `variant="muted"`, не `outline`.

## Components

Primitives + Foundation, собранные в конкретную повторяемую сущность. Раздел организован по семействам — по вопросу «что мне нужно показать», а не по происхождению кода.

### Семейство Card — «карточка чего-то»

В Storybook — папка `Components/Card/*`. В коде это НЕ один компонент: `Card` — вендорная оболочка, остальные — независимые композиции на ней (склеивать их в один compound-API нельзя — `card.tsx` вендорен и не редактируется).

| Задача | Бери | Почему |
|---|---|---|
| Универсальная оболочка (Header/Content/Footer) | `Card` 🔵 | База всего семейства. На тёплом фоне страницы осознанно выбирай между ней и `Surface variant="panel"` (см. гайдлайн ниже). |
| Совет/инсайт в side-rail | `AdviceCard` 🟢 `icon, tone("ink"\|"accent"), value?, text, note?` | Для `TitledRow`'s `side`. `tone="accent"` — редкость (новый рекорд), не дефолт. Не путать с `Alert` (плоское уведомление, без слота под значение) и `Empty` (про отсутствие данных). |
| Карточка-метрика (число + тренд + мини-график) | `MetricCard` 🟢 `label, value, delta?, tone?, trend?, spark?, sparkColor?` | Форма — Steep (число+дельта слева, спарклайн справа, точка на конце). Строчный вариант без карточки — `MetricRow`. |
| Карточка с графиком (заголовок + легенда/стат + чарт) | `ChartCard` 🟢 `title, sub?, aside?, children` | Самая частая композиция dashboard-страницы. Внутрь — любой `Components/Chart/*` как children. Не собирай Card+заголовок+чарт руками. |
| Двухслойная «факт → совет» (наблюдение на бумаге + рекомендация на тинте) | `LayeredCard` 🟢 `children, footer?` | Анатомия `.rec`/`.up4-card` страницы разбора диалога: opaque-бумага с фактом внутри тонированной обёртки (rgba(0,0,0,.03), 8px рамка), совет/скрипт — в `footer` на тинте. Не собирай вложенные фоны руками. Жанр — «факт + обвязка»: наблюдение/цитата на бумаге, а к нему разбор/скрипт/оценка в `footer` (хайлайт-карточка библиотеки, инцидент с рекомендацией — двухзонность подтверждена Maze Highlight). **Эмфаза фиксирована: главное (факт/хайлайт) — на бумаге, вторичное (сноска, «разобрать», мелкий CTA) — в `footer`. Не наоборот** (тинт ~3%, сам по себе ничего не объясняет). **Тест применимости — ЧТО главное в секции:** если это НАБЛЮДЕНИЕ (что произошло) с приложенным разбором — `LayeredCard` (например «Что улучшить» — факт про диалог + совет). Если суть секции — ДЕЙСТВИЕ (готовый скрипт + «потренировать» как основной пейлоуд, а не сноска) — двухзонность лишняя, бери обычную `Surface` с явной иерархией. Разные секции разбора отличаются именно этим, не путать по похожим названиям. |
| Карточка сущности (аватар+имя+тренд+теги+метрики+CTA) | гайдлайн ниже | Не компонент — но правила сборки жёсткие, см. ниже. Ингредиенты: `SemanticAvatarFallback` + `StatusBadge` + `MetricRow` + `Separator` + `Button`. **Оценка в карточке — цветной `Text` (score-band), НЕ `ScorePill`:** пилюля — язык СПИСКОВ/строк, в карточку сущности не кладётся (DESIGN.md → «Два языка оценки»). |

**Гайдлайн: карточка сущности.** Правила:
- Разделитель — между КРУПНЫМИ смысловыми блоками, не перед каждым элементом. Три блока, не пять: (шапка + статус-тег вместе, это одно «сейчас с этим человеком») → (метрики) → (CTA). Тег после шапки — без своего отдельного `<Separator />`, это тот же блок, ему нужен только `gap`. **И обратное правило: разделитель существует только МЕЖДУ двумя непустыми блоками** — если у карточки нет метрик (управляющий без цифр), их разделители уходят вместе с ними; две линии вокруг пустой зоны — не «сохранение структуры», а мусор. Равнять высоты карточек пустыми зонами тоже не нужно — `Grid` с `align:start` этим управляет сам.
- `Surface variant="panel"` даёт тёплый кремовый тон (cream-1); `variant="paper"` — чистый белый: осознанный выбор, не default по инерции.
- **Имя человека — всегда `body`/600, где бы он ни показывался** (карточка, компактная строка инсайтов, список): источник держит один размер (`--text-body`) и в `.ename`, и в `.iline .nm`. Не даунскейлить имя до `caption` в «маленьких» рядах — компактность даёт вторая строка (`footnote`), не усушка имени.
- Тон аватара и тон бейджа — из одной роли, и оба — **мягкая пара** (`color.<tone>Soft`/`SoftForeground`). Яркая пара (`color.<tone>`) — язык score-гейджа (`RadialChart`), не аватара. **Парность «аватар = бейдж» — только когда общий источник это РОЛЬ.** Если ролевого цвета у сущности нет (рядовой сотрудник без управляющей роли), а есть лишь СТАТУС («низкая оценка», «требует внимания») — аватар нейтральный (`muted`), а статус-бейдж несёт свой статусный тон САМ, отдельно. Статус на аватар не зеркалим: иначе «серый человек» превращается в «тревожно-красного человека», хотя красное — про его метрику, а не про него.

### Семейство Chart — все графики

В Storybook — папка `Components/Chart/*`. Все — настоящий Recharts через shadcn's `ChartContainer`; grid/оси — `color.input`, никогда `color.border` (тот прозрачный по правилу).

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `BarChart` / `LineChart` / `AreaChart` | `data, xKey, series: {key,label,color}[], height?, showLegend?` (+`stacked?` у Area) | Базовые типы. `showLegend` по умолчанию `series.length > 1`. |
| `DonutChart` | `data: {key,label,value,color}[], height?, showLegend?` | Категориальная разбивка (Pie, `innerRadius:"60%"`). |
| `DistributionChart` | `groups: {name,total,segments:{label,count,widthPct}[]}[], rowHeight?` | Горизонтальный stacked-bar на группу; недостающие до 100% сегменты дорисовываются приглушённым `rest`. |
| `ScatterChart` | `series: {key,label,color,data:{x,y}[]}[], xLabel?, yLabel?, height?, showLegend?` | Корреляция двух числовых величин. |
| `RadialChart` | `value, max?(=100), color, trackColor?, size?, label?` | Круговой гейдж («ScoreRing»). **`value` — в СВОЕЙ шкале, не пере-масштабируй в 0–100:** «3,8 из 10» → `value={3.8} max={10}`, а не `value={38}`. Задавай `max` так, чтобы `value` и центральный `label` были одним числом — иначе они разъезжаются (был foot-gun). |
| `ComposedChart` | `data, xKey, series: {key,label,color,type:"bar"\|"line"\|"area",axis?:"left"\|"right"}[], height?, showLegend?` | Смешанные типы серий. **`axis:"right"` не декоративен**: серии на сильно разных масштабах на общей оси сплющиваются в прямую линию. |
| `Sparkline` | `data: number[], color, width?, height?, showEndDot?` | Мини-тренд внутри строки/карточки — без осей/сетки/тултипа, фиксированный размер. **Без `ResponsiveContainer` намеренно** — процентный размер внутри auto-height Grid/Flex-строки не на что мерить (баг Recharts). |

`recharts` — реальная, объявленная зависимость этого пакета. Если готовых чартов не хватает — можно писать напрямую на `recharts`, но объявляй его явной зависимостью в своём `package.json`, не рассчитывай на хойстинг node_modules (сегодня долетает транзитивно — это случайность дерева, не гарантия).

### Тулбары и сегмент-контролы — какой когда

Похожий «сегментированный» ряд дают несколько компонентов — роли разные:

- **Переключение состояния/периода** (Часы/Дни/Недели, вкладкоподобное) → `ToolbarToggleGroup` + `ToolbarToggleItem` (roving focus, aria-состояние «выбрано»); голый примитив под ним — `ToggleGroup`.
- **Ряд независимых действий-кнопок** (два спаренных icon-баттона, сортировка) → `ButtonGroup` (Primitives) — каждая кнопка делает действие, ничего не «выбрано».
- **Главное действие + меню его вариантов** (Копировать рекап ▾, Синхронизировать ▾) → `SplitButton` (Components). **Не собирай из `ButtonGroup`+`DropdownMenu`**: у `ButtonGroup` pill-оверрайд выключен, он фьюзит сегменты в 14px-ПРЯМОУГОЛЬНИК — язык toggle/фильтров, инородный нашим pill-кнопкам (проверено на meeting-recap: split из группы читался разъединённым rect'ом рядом с pill-CTA). `SplitButton` держит цельную pill-форму.
- **Переход между разделами/страницами** → `NavigationMenu` (Components) — настоящие `<a href>` в `<nav>`-ландмарке: URL/history, открытие в новой вкладке, «navigation» для скринридера. Правило одной строкой: **меняешь данные на экране — ToggleGroup, меняешь экран — NavigationMenu**; «абстрактный» ToggleGroup вместо навигации теряет весь контракт ссылки.

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Toolbar` + `ToolbarGroup` + `ToolbarToggleGroup` + `ToolbarToggleItem` + `ToolbarSeparator` + `ToolbarButton` 🟢 | `Toolbar({orientation, bare})`; `ToolbarToggleGroup({value, onValueChange})` | На `radix-ui` Toolbar напрямую (у shadcn нет рецепта). **`bare=true` обязателен**, если внутри ровно одна самоокрашенная группа — иначе пилюля-в-пилюле. |
| `SplitButton` 🟢 | `children, icon?, onClick?, menu, menuLabel, variant?("secondary"дефолт\|"outline"\|"ghost"\|"default"), align?, disabled?`; `menu` — ReactNode из `DropdownMenuItem`/`DropdownMenuRadioGroup` | Главное действие + ▾-меню его вариантов, в pill-форме. Заливка `secondary` по умолчанию: два сегмента одного фона читаются как один контрол, шов их делит. `outline` даёт разъединённый вид — есть для гибкости, но не дефолт. Внешние углы pill, внутренние срезаны (инлайн-радиус бьёт brand-pill правило). |

### Списки и композиционные оболочки (shadcn/vendored)

`Card` (см. семейство Card), `Empty`, `Tabs`, `Alert`, `AlertDialog`, `DropdownMenu`, `Pagination`, `Table`, `Toaster` (sonner), `Item`, `NavigationMenu`. Композиционные оболочки со слотами — поэтому Components, не Primitives. Граница с Primitives-vendored: `Dialog`/`Select`/`Popover`/`ScrollArea` остаются примитивами — многочастная разметка, но ОДНО взаимодействие (открыть модалку, выбрать одно, показать флоат, проскроллить); оболочки же структурируют контент СЛОТАМИ (Header/Content/Footer у Card, Media/Title/Actions у Item). Тест: если children кладутся внутрь как есть, без раскладки по слотам, — это примитив, какой бы многочастный DOM ни был у него внутри. `NavigationMenu` (8 слотов, выпадающие панели, свой viewport) переехал сюда именно по этому критерию. API — см. shadcn-документацию.

- **`Alert`** — статичная заметка (заголовок + текст), ось одна: `variant` (`default`/`destructive`). НЕ для продуктового баннера с кнопкой действия («Распознано N голосов» + «Разметить») — это рецепт `Surface variant="muted"` + `Button` (см. правило у `Surface`); и НЕ для «требуют внимания» по метрикам — это язык `accent`, а `destructive` только для жёстких ошибок (правило danger в `DESIGN.md`).
- **`AlertDialog`** — блокирующее подтверждение разрушительного действия. Ровно то место, куда указывает правило про `danger` в `DESIGN.md`: кнопка подтверждения — единственное легитимное место для danger-filled кнопки. Для нейтральных модалок — `Dialog`, не он.

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `Item` + `ItemGroup`/`ItemSeparator`/`ItemMedia`/`ItemContent`/`ItemTitle`/`ItemDescription`/`ItemActions`/`ItemHeader`/`ItemFooter` | Governed строка списка: иконка/медиа + заголовок + описание + trailing-действия. `variant("default"\|"outline"\|"muted")`, `size("default"\|"sm"\|"xs")`. | Любой список строк одной формы — диалоги, уведомления, activity feed. Не собирай flex-строку + свой `<hr>` под этот же паттерн. `variant="outline"` — см. предупреждение про `color.border` выше. |
| `CollapsibleGroup` 🟢 `title, count?, extra?, defaultOpen?, open?, onOpenChange?, children` | Сворачиваемая группа с кликабельной шапкой: заголовок + счётчик + слот под бейдж + поворачивающийся шеврон. | Группировка карточек по адресу/категории. Контролируемый режим (`open`/`onOpenChange`) — для «Свернуть всё». Длинный список внутри — оберни в `ScrollArea`, не самодельный overflow. |

⚠️ **Известный пробел:** `Toaster` (sonner) использует `next-themes`, но `ThemeProvider` не смонтирован — тема тостов всегда резолвится в `"system"`. Не чинилось, низкий приоритет.

⚠️ **Известный пробел (баг апстрима shadcn, проверено в их registry):** `Progress` не передаёт `value` в Radix Root — визуально полоса правильная, но для скринридеров ЛЮБОЕ значение читается как indeterminate (нет `aria-valuenow`). Локально не патчим (vendored-исходники не редактируются); если критично на конкретном экране — ставь `aria-valuenow`/`aria-valuemax` пропами прямо на `<Progress>` (они долетят до Root через `...props`).

## Sections

Секции — намеренно немногочисленные (стоящее правило: `Sections` остаётся компактным, а не превращается во вторую копию `Components`; странично-специфичный блок без повторяющегося прецедента на нескольких реальных экранах продукта не пакуется в новую секцию раньше времени — он остаётся кодом конкретной страницы в потребляющем приложении).

| Компонент | API | Когда брать / не брать |
|---|---|---|
| `SidebarNav` | `active?, items (обязателен, NavItem[]), avatarInitials?, avatarTitle?, showSettings?, clientLabel?, clientInitial?` | Левый рейл приложения. `items` — обязательный проп, без дефолта (пакет обслуживает несколько продуктов/клиентов Артефакта — ни один конкретный набор пунктов не может быть «тем самым» дефолтом). `clientLabel` — опциональный чип «какой это клиент»; чип не рендерится вообще, если его не передать, никогда не показывает чьё-то конкретное имя по умолчанию. `EXAMPLE_NAV_ITEMS` экспортирован как пример/фикстура, не как рабочий дефолт. |
| `AssistantDock` | `title?, intro?, chips?(ChipQuestion[]), fallbackAnswer?, placeholder?` + ref-handle `{ask(question, answer?)}` | Плавающий AI-ассистент. Настоящий стейт-мачин (`idle → open → orb`), не статичный снэпшот — `orb` включается, если пользователь скроллит вниз и закрывает панель (`window.scrollY > 120`). `intro`/`fallbackAnswer` — placeholder-дефолты без привязки к конкретному продукту/клиенту, передавай реальный текст явно. Единственный наравне с `SidebarNav` «органический» (bespoke) компонент — блюр-панель/градиентная кнопка/спиннер-иконка — реальный CSS в `AssistantDock.css`, не выражен через Foundation-примитивы (не всё честно сводится к токенам без выдумывания). **Ref-handle `ask(question, answer?)`** — способ, которым остальная страница вызывает ассистента с готовым вопросом (плашка «Назначить профиль», каждый `SparkLink`, «Потренировать с ассистентом» в допродажах). |
| `PageHero` | `title, crumbs?, badge?, sub?, lead?, action?, fullWidth?` | Страничная шапка В ПОТОКЕ контента (Deel/Neon): h1 слева + CTA справа одной строкой, `crumbs` (навигация назад) — всегда НАД заголовком, пилюли статуса (`badge`) — ПОД ним, `sub` — мета-строка фактов, `lead` — короткая постановка абзацем (Docusign). **Правило выбора против `PageHeader`:** карточки стоят сразу рядом с шапкой → рейл (`PageHeader`); списочная или детальная страница → hero. |
| `PageHeader` | `title, meta?, action?` | Шапка страницы: заголовок в рейле `TitledRow` + мета-строка + опциональный CTA справа. Это единственное легитимное использование рейла НЕ по правилу «≥2 карточек» — тут в рейле заголовок СТРАНИЦЫ, не секции. `action` — обычно то самое единственное главное действие экрана (см. правило про дефолтную кнопку в `DESIGN.md`). |

**Жанровые правила страниц** (выбор скелета по вопросу, на который отвечает страница):

| Задача | Бери | Почему |
|---|---|---|
| Досье сущности (HR-профиль, карточка объекта) | идентити-рейл слева + разделы табами | Норма жанра: Aboard, Jira, Confluence. Рейл свойств — язык досье. |
| Одно-сценарная аналитическая страница («почему просела оценка») | `PageHero` + лента секций `TitledRow`, без табов | Страница отвечает на ОДИН вопрос — рейл и табы дробят повествование (контрольный тест B1). «Паспорт»-рейл здесь читается надуманным — это жанр тикета (Jira/Linear). |
| Страница-отчёт (research-репорт, протокол) | summary/вердикт-карта ПЕРВОЙ, хронология — карточками с шапками-шагами, evidence — пилюлями | Канон Dovetail (Summary first) и Maze (SUMMARY → нумерованные DETAILS). Мета — строкой фактов в шапке, не боковым рейлом. |
| Переключение жанра контента vs фильтры выборки | `Tabs` в шапке; фильтры — `Search` + `FilterSelect` одной строкой над гридом | Разные роли = разная форма (Substack, Xero, Dribbble, Dovetail). Два `ToggleGroup` рядом для разных ролей — анти-паттерн. |
| Сравнение сущностей | метрики/атрибуты → `Table`, сущности — колонками с карточками в шапках; динамика → ОДИН график с наложением серий | Норма: Zillow/Hootsuite (таблица), Amplitude/Mixpanel (наложение). Парные карточки — жанр сравнения ОПЫТА (Shop, Chatbase), не метрик; дельта-бейджи в каждой строке — не норма; для 2 сущностей с коротким списком допустимы зеркальные локапы. |
